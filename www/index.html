// why engram

Memories Claude writes, but you can't see.

Claude Code is quietly building a memory of how you work — one markdown file at a time, buried under ~/.claude/projects/<project>/memory/. It works, until you actually want to use it.

the problemMemories are scattered one folder per project — awkward to read raw in a pager.
what engram doesOne searchable two-pane UI across every project.
the problemThe team-useful ones are stuck on one laptop, with no way to share them.
what engram doesPromote one and your teammates pull it over plain git.
the problemEditing the raw files by hand risks corrupting what Claude reads.
what engram doesRead-only by default — engram only ever adds its own frontmatter.

Not a replacement for committing a CLAUDE.md to your repo — engram covers the gap that can't: cross-project memories, a personal-vs-team layer, and a real UI. It pairs with Claude Code's own /memory and /memory-check commands, adding the cross-project view and team layer those don't.

// the moat

Your memory, not their server.

Every assistant now keeps a memory of how you work. The real question is where it lives — and whether you can read, move, and version it.

auto-learned memorywhere it livesyours to move?
Claude Code + engram your disk · plain Markdown yes — it's just files in git
GeminiGoogle account · serverno — vendor-locked
GrokxAI account · serverno — vendor-locked
CopilotGitHub account · serverno — vendor-locked

Claude Code is the outlier: it writes its memory to ~/.claude as Markdown you own. engram is the fast UI and the git layer on top — so your team's hard-won context stays portable, reviewable, and never trapped behind someone's login.

Scope: the memory an assistant learns on its own. You-authored instruction files like GEMINI.md or copilot-instructions.md can live locally too — it's the auto-memory that's locked to a vendor's account.

// team sharing

Promote a memory. Your team pulls it.

The shared store is one git repo your whole team reads and writes. engram keeps a managed clone and shells out to git for every sync — no servers, no engram-level auth, nothing new to trust. Personal memories never leave your machine unless you promote them.

1 · onceengram init-team <git-url>
→
2 · in the TUIselect & promote — commit + push
→
3 · teammatepull — lands where Claude reads

Setup is one command, once. Day to day, promote and pull are keystrokes inside the TUI — engram stays a no-arg tool.

// the day-to-day

Built for living in the terminal.

◧

Two-pane TUI

A searchable list beside a fully rendered markdown preview. Hit / to fuzzy-filter across titles, projects, and types — Glamour-styled and keyboard-driven.

✎

Edit in $EDITOR

Press e to open the selected memory in your editor, then reload in place. n to create, d to delete — the MEMORY.md index stays in sync.

▦

Group your way

Toggle between by-project and by-type with g; filter to a single type with t. Color-coded badges for user, feedback, project, reference.

⌘

Command palette

ctrl+p opens to two guides: type / for /memory, /plans, and /files (your CLAUDE.md + MEMORY.md, read-only), or @ for the assistant.

✦

@Claude repairs drift

Hand malformed frontmatter, broken [[links]], or files stranded by a renamed project to an interactive Claude Code session, seeded with your memory health. engram reloads when it exits.

◑

Live theming

Classic Dark, Tokyo Night, Gruvbox, Nord, Dracula. Switch with 1-5; your choice persists under XDG config.

// zero setup

It already knows where to look.

engram scans ~/.claude/projects/*/memory/ and parses both shapes Claude Code writes — YAML frontmatter and plain markdown with a MEMORY.md index. Nothing to configure.

  • Reconciles a drifted index with R
